História e Contextualização

Fundação

Criado pela empresa alemã ArangoDB GmbH em 2011. Idealizado por Claudius Weinberger e Frank Celler.

Versão

Primeira versão estável lançada em 2012. Atualmente na versão 3.11 (2024).

Nome

"Arango" foi inspirado no termo "araneus" (aranha em latim), simbolizando a natureza de múltiplas conexões dos grafos.

Popularidade

Possui mais de 11.000 estrelas no GitHub e uma comunidade global ativa.

Evolução Contínua

Implementação de recursos avançados como ArangoSearch, SmartGraphs e melhorias na performance e escalabilidade.

    Escalabilidade e Arquitetura

  • Escalabilidade horizontal simplificada sem comprometer desempenho;
  • Eliminação da necessidade de sincronização entre diferentes bancos de dados.

    • Casos em que NÃO é recomendado

    ❌

    Quando Não usar o ArangoDB

  • Aplicações bancárias e sistemas financeiros legados com forte dependência do modelo relacional;
  • Aplicações que requerem esquema rígido e forte consistência em todas as operações;
  • Casos de uso que exigem joins complexos entre muitas tabela;s
  • Quando o modelo relacional é suficiente e a equipe já domina SQL;
  • Ambientes com recursos de hardware muito limitados.

    Estrutura e Funcionamento Interno

    📄

    Como os dados são armazenados

  • Multi-modelo: combina armazenamento de documentos, grafos e chave-valor
  • Documentos são armazenados em formato JSON/BSON estendido
  • Grafos são implementados com coleções de documentos para vértices e arestas
  • Usa o motor de armazenamento "RocksDB" por padrão (otimizado para SSD)
  • Sistema de cache em camadas

    • Estruturas básicas e elementos-chave

    🗂️

    Coleções

  • Agrupam documentos semelhantes (equivalente a tabelas);
  • Coleções de documentos: Armazenam documentos regulares;
  • Coleções de arestas: Armazenam conexões entre documentos (para grafos).
    • 📄

    Documentos, índices e grafos

  • Documentos: Objetos JSON com identificador único (_id, _key, _rev);
  • _id: identificador completo (coleção/chave);
  • _key: identificador único dentro da coleção;
  • _rev: revisão para controle de concorrência (MVCC);
  • _from, _to: específicos para documentos em coleções de arestas;
  • Índices de diferentes tipos para otimizar consultas (hash, skiplist, geo, fulltext, etc.);
  • Grafos compostos por coleções de vértices e arestas.
    • 🔑

    AQL (ArangoDB Query Language)

  • Linguagem declarativa para consultas em todos os modelos
  • Inspirada em SQL mas com extensões para documentos e grafos
  • Suporte a operações CRUD completas
  • Funcionalidades avançadas: funções geoespaciais, funções de texto, agregações
  • Otimizador de consultas com múltiplas estratégias

    • Arquitetura

    📄

    Distribuição (Distribuição de Dados)

  • ArangoDB é distribuído por natureza: Ele pode ser executado em um cluster com vários servidores;
  • Divide os dados em partes (shards): Cada coleção é dividida em pedaços chamados "shards", que são distribuídos entre diferentes nós;
  • Coordenadores: recebem as requisições dos usuários e decidem onde os dados estão;
  • DB-Servers: armazenam os shards (dados reais).
    • 📄

    Replicação (Cópia de Dados)

  • Alta disponibilidade com cópias dos dados: Cada shard pode ter réplicas (cópias) armazenadas em outros servidores;
  • Failover automático: Se um servidor falhar, outro servidor com a réplica pode assumir automaticamente;
  • Modo síncrono ou assíncrono:
  • Síncrono: garante que os dados estejam sempre consistentes entre as réplicas;
  • Assíncrono: tem desempenho mais rápido, mas pode haver atraso na sincronização dos dados;
    • 📄

    Escalabilidade (Crescimento do Sistema)

  • Escalabilidade horizontal: É possível adicionar mais máquinas ao cluster para aumentar a capacidade;
  • Balanceamento de carga automático: O ArangoDB distribui os shards entre os servidores para evitar sobrecarga;
  • Boa performance com grandes volumes de dados: Mesmo com muitos dados ou consultas complexas, o banco mantém bom desempenho.

    • Teorema CAP e Propriedades

    ArangoDB é principalmente um sistema CP (Consistência e Tolerância a Particionamento). No modo cluster, prioriza consistência sobre disponibilidade em caso de particionamento de rede. No modo single-server, atua como um sistema CA.

    Exemplos de Consultas AQL

    Inserção de Dados 
    // Inserir documentos na coleção 'usuarios'
INSERT {
  nome: "João Silva",
  idade: 28,
  cidade: "São Paulo",
  interesses: ["tecnologia", "futebol", "música"]
} INTO usuarios

// Inserir múltiplos documentos
FOR user IN [
  {nome: "Maria Santos", idade: 32, cidade: "Rio de Janeiro"},
  {nome: "Pedro Costa", idade: 25, cidade: "Belo Horizonte"}
]
INSERT user INTO usuarios
    Consultas e Filtros 
    // Buscar usuários por idade
FOR user IN usuarios
  FILTER user.idade >= 25
  RETURN {nome: user.nome, idade: user.idade}

// Buscar com múltiplos critérios
FOR user IN usuarios
  FILTER user.cidade == "São Paulo" AND user.idade < 30
  SORT user.nome ASC
  RETURN user
    Consultas em Grafos 
    // Criar relacionamento de amizade
INSERT {
  _from: "usuarios/joao",
  _to: "usuarios/maria",
  tipo: "amizade",
  desde: "2023-01-15"
} INTO amizades

// Encontrar amigos de amigos
FOR user IN usuarios
  FILTER user.nome == "João Silva"
  FOR friend IN 1..2 OUTBOUND user amizades
    RETURN DISTINCT friend.nome
    Agregações e Análises 
    // Estatísticas por cidade
FOR user IN usuarios
  COLLECT cidade = user.cidade WITH COUNT INTO total
  RETURN {
    cidade: cidade,
    quantidade: total,
    percentual: ROUND(total * 100 / LENGTH(usuarios), 2)
  }

// Usuários mais conectados
FOR user IN usuarios
  LET connections = (
    FOR v IN 1..1 OUTBOUND user amizades
      RETURN 1
  )
  RETURN {
    nome: user.nome,
    amigos: LENGTH(connections)
  }
  SORT LENGTH(connections) DESC
  LIMIT 10

    Web Interface

    REST API

    API RESTful completa para todas as operações CRUD, administração de coleções e execução de consultas AQL via HTTP.

    ⚙️

    Foxx Framework

    Framework JavaScript para criar microserviços customizados diretamente no banco, com roteamento e middleware integrados.

    Exemplo Foxx Microservice 
    // main.js - Serviço Foxx para API de usuários
const createRouter = require('@arangodb/foxx/router');
const db = require('@arangodb').db;

const router = createRouter();

// Endpoint para buscar usuários
router.get('/usuarios', function (req, res) {
  const users = db._query(`
    FOR user IN usuarios
      RETURN {
        id: user._key,
        nome: user.nome,
        cidade: user.cidade
      }
  `).toArray();
  
  res.json(users);
});

// Endpoint para criar usuário
router.post('/usuarios', function (req, res) {
  const userData = req.body;
  const result = db.usuarios.save(userData);
  res.json({success: true, id: result._key});
});

module.context.use(router);

    Conclusão e Discussão

    ✨

    Pontos Fortes

    • Versatilidade para múltiplos modelos de dados em um único sistema;
    • Linguagem de consulta poderosa e intuitiva;
    • Excelente desempenho em consultas de grafo e documentos complexos;
    • Interface web completa para administração e monitoramento;
    • Escalabilidade horizontal eficiente;
    • Comunidade ativa e bom suporte;
    • API HTTP RESTful abrangente.
    ⚠️

    Limitações

    • Curva de aprendizado inicial para dominar o AQL;
    • Menor adoção no mercado comparado a MongoDB ou Neo4j;
    • Menos profissionais especializados disponíveis no mercado;
    • Ecossistema de ferramentas menos desenvolvido que concorrentes mais estabelecidos;
    • Consumo de recursos relativamente alto em comparação com bancos mais especializados.

    Recapitulação dos pontos principais

    ArangoDB é um banco de dados multi-modelo que integra de forma nativa os modelos de documentos, grafos e chave-valor em um único sistema. Uma de suas principais características é a linguagem de consulta unificada AQL (ArangoDB Query Language), que permite consultar e manipular todos esses modelos de dados com uma sintaxe consistente e poderosa. O ArangoDB também oferece escalabilidade horizontal por meio de clusters e sharding automático, facilitando a distribuição e o gerenciamento de grandes volumes de dados. Em termos de consistência, o banco implementa o modelo CP do teorema CAP, priorizando a consistência e a tolerância a particionamento em ambientes distribuídos. Dessa forma, o AQL possibilita uma abordagem unificada para lidar com diferentes tipos de dados, simplificando o desenvolvimento de aplicações complexas.
